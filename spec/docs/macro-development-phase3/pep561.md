# PEP 561 対応仕様

## 1. 目的

インストール済みの `nyxfw` から import される `nyxpy` を、型検査器とエディタが typed package として扱えるようにする。マクロ実装者と AI agent が `nyxpy.framework.*` の型ヒントを補完・静的解析で利用できる状態を作る。

## 2. 現状

| 項目 | 状態 |
|------|------|
| `src\nyxpy\py.typed` | 追加済み |
| 型ヒント | 公開 API に存在するが、配布 package として typed であることを示していない |
| stub package | なし |
| build backend | hatchling |

## 3. 判断

`src\nyxpy\py.typed` を追加する。NyX は inline type hints を持つ package として扱い、別途 `.pyi` stub package は作らない。

## 4. 実装仕様

| 作業 | 内容 |
|------|------|
| marker 追加 | 空ファイル `src\nyxpy\py.typed` を追加する |
| package 対象 | `src\nyxpy` 配下に置き、wheel に `nyxpy\py.typed` として含める |
| 公開範囲 | 型情報の主対象は `nyxpy.framework.*` とする |
| 非対象 | GUI / CLI の内部実装は、マクロ実装者向けの安定 API として扱わない |

## 5. 検証仕様

```powershell
uv build
$wheel = Get-ChildItem dist\*.whl | Select-Object -First 1
python -m zipfile --list $wheel.FullName
uv run ty check src\nyxpy --output-format concise --no-progress
```

検証時は wheel 一覧に `nyxpy\py.typed` が含まれることを確認する。

## 6. 型検査 baseline

`ty` を開発依存に追加し、PEP 561 公開後に利用者へ見える型ヒントの診断を取得する。現在は `src\nyxpy` 全体が 0 diagnostics になっているため、`ty check` を CI gate として扱う。

初回確認で目立つ診断は次の種類である。

| 分類 | 例 |
|------|----|
| `None` を許容していない型注釈 | `Command.capture()` の `crop_rect=None`、通知画像の `img=None` |
| 動的に追加した class 属性 | `LStick.UP`, `LStick.DOWN`, `RStick.CENTER` |
| 外部ライブラリ stub との差分 | PySide6 enum、OpenCV / PaddleOCR の戻り値 |
| 例外 wrapper の `**kwargs` 型 | `FrameworkError.__init__` へ渡す `object` |
| 設定辞書の union 型 | `SettingsSchema` 周辺の nested dict access |

公開 API から順に型注釈を修正し、`src\nyxpy\framework\core` と `src\nyxpy` 全体は 0 diagnostics に到達済みである。`tests` / `examples` は必要に応じて別 phase で対象範囲を広げる。

## 7. 注意点

`py.typed` を追加すると、既存の不正確な型ヒントも利用者側の型検査に現れる。公開 API の型ヒントと docstring に矛盾を見つけた場合は、型公開を避けるのではなく、該当 API の型ヒントを修正する。
